# Plan: Investigate & fix flakey karma tests in `public/test`

Five distinct intermittent failure signatures appear across the 47 CI logs. Most are timing/race-condition failures rather than logic bugs — the tests rely on hard‑coded timeouts and event ordering that occasionally lose a race under CI load. Two low-frequency signatures (D, E below) are assertion mismatches caused by async state propagation, not hard timeouts. The plan instruments the tests to confirm root cause, then replaces timing-based synchronization with deterministic waits.

**The five failure signatures**
1. **(Sig C)** `Latest widget > searches articles` — `Cannot read properties of undefined (reading '$component')` at [latest.articles.spec.js](public/test/spec/latest.articles.spec.js#L28). `getAutocomplete` does `ko.contextFor(host.container.querySelector('autocomplete').firstChild).$component`. The test resolves on the first `widget:load` event, but `widget:load` is emitted by *every* `BaseWidget` after 20ms ([base-widget.js](public/src/js/widgets/base-widget.js#L9)), so the `<autocomplete>` element's `firstChild` may not be rendered yet → `contextFor` returns `undefined`. The trace logs confirm this: lines like `widget:load emitted by <non-autocomplete widget> … DOM autocomplete: null` show other widgets winning the race.
2. **(Sig A)** `Config Front > create fronts and collections` — `Timeout waiting for condition` from `wait.condition` (5000ms default) at [wait.js](public/test/utils/wait.js#L33). **Root cause confirmed (order-dependent global-singleton leak):** the failing `changeImageUrl` step calls `validateImageSrc` → `stripImplementationDetails` → `grid()`. [utils/grid.js](public/src/js/utils/grid.js#L7) lazily builds a module-level memoized singleton `grid.gridInstance`, reading `vars.model.state().defaults.baseUrls.apiBaseUrl` **only on first construction**. The `config.front.spec` model had `defaults: { editions: [] }` with **no `baseUrls`**, so when this step is the first in a run to call `grid()`, construction throws `Cannot read properties of undefined (reading 'apiBaseUrl')` → error branch → `alert()` → `saveProps`/`persistence.front.update` never called → 5s timeout. When an earlier spec warmed the singleton with a baseUrl-bearing model, the cached instance is reused and the test passes — hence the flake depends purely on karma spec order. The image itself loads in **0ms** (`fetchImage onload after 0 ms`), disproving the slow-load idea.
3. **(Sig B)** `Breaking News > Alerts the user before launch` **and** `Breaking News > Allows the same alert to be sent multiple times` — `edit action timeout, endpoint /edits` from the 2000ms hard timeout in [base-action.js](public/test/utils/actions/base-action.js#L13). Both tests live in [breaking-news.spec.js](public/test/spec/breaking-news.spec.js) and share the same `/edits` request-timeout race; "Allows the same alert…" (11) actually flakes more often than "Alerts the user before launch" (6).
4. **(Sig D)** `Collections > copy to clipboard` ([collections.spec.js](public/test/spec/collections.spec.js#L324)) — `Expected 'Open Source' to be 'Nothing happened for once'`. Copies `latest().trail(5)` to the clipboard then asserts the clipboard's first trail headline. The wrong article lands in the clipboard, i.e. the latest list is not fully/stably populated when `trail(5)` is resolved, or clipboard state leaks across tests. **This is an assertion flake, not a hard timeout.**
5. **(Sig E)** `Visited articles > marks an article visited from clipboard as visited in visited articles list` ([visited.articles.spec.js](public/test/spec/visited.articles.spec.js#L31), assertion at line 52) — `Expected 1 to be close to 0.6`. Copies `latest().trail(1)`, opens the link in the clipboard, then expects the matching trail in the latest list to be dimmed (`opacity ≈ 0.6`) but it is still `1`. The "visited" state hasn't propagated back to the latest list when the assertion runs. **Also an assertion flake driven by async state propagation, not a hard timeout.**

**Hypotheses (to confirm in Phase 2)**
- **H1 (Sig C — searches articles):** `getAutocomplete` resolves on the *first* `widget:load` event, but that event is emitted by every `BaseWidget` 20ms after construction ([base-widget.js](public/src/js/widgets/base-widget.js#L9)). When a non-autocomplete widget wins the race, `querySelector('autocomplete').firstChild` is still null/un-rendered, so `ko.contextFor(...)` returns `undefined` and reading `.$component` throws. **Confirmed by the trace logs** (`widget:load emitted by <non-autocomplete widget> … DOM autocomplete: null`).
- **H2 (Sig A — create fronts and collections):** ~~A CAPI backfill step's `waiting.check` predicate occasionally exceeds `wait.condition`'s 5000ms budget.~~ **DISPROVEN.** The failing step is `changeImageUrl`, not a backfill step. Two pieces of evidence: (1) raising the budget to 10000ms (uncommitted edit on line 241, since reverted) turned the clean 5s `condition timeout` into a full 10s Jasmine hang — i.e. `persistence.front.update` was still never called after 10s, so it is **not** a "needs slightly more than 5s" race; (2) the original trace shows `predicate result: false` after the *entire* 5s elapsed. **Refined hypothesis H2':** the `changeImageUrl` step depends on a real `new Image()` load of the `square.png` fixture; under CI load it is slow or fires `onerror`, and the error branch in the front.js subscription calls `alert()` instead of `saveProps`, so `persistence.front.update` is never invoked. Instrumentation added (front.js START/SUCCESS/ERROR + fetchImage onload/onerror timing) to confirm which of: slow `onload`, `onerror`, or `onload` never firing.
  - **Important side effect discovered:** the reverted `, 10000` budget edit also spawned a confusing cross-test error — `Uncaught TypeError: Cannot read properties of undefined (reading 'any')` — because the leaked `wait.condition` polling interval kept evaluating `persistence.front.update.calls.any()` after `afterEach` reset the spy. This is a latent hazard in [wait.js](public/test/utils/wait.js) `condition` (it never clears its interval on the host test ending) worth hardening separately.
- **H3 (Sig B — both breaking-news tests):** The `/edits` POST either (a) is never triggered by the paste→edit chain within 2000ms, or (b) is triggered but its `onAfterComplete` resolution (an inner 20ms `setTimeout`) is cleared/lost during teardown — so `waitForRequest` rejects. The same race affects both `Alerts the user before launch` and `Allows the same alert to be sent multiple times`. Expected to confirm by tracing whether the mockjax interceptor fired and whether `lastRequest` was captured before the timeout.
- **H4 (Sig D — copy to clipboard):** `latest().trail(5)` is resolved before the latest list has stably rendered all 5 trails (or clipboard state leaks from a prior spec), so a different article (`'Open Source'`) is copied instead of the expected `'Nothing happened for once'`. Expected to confirm by tracing the latest list contents/length at copy time and whether the clipboard was non-empty at test start.
- **H5 (Sig E — visited articles):** After `openLink()` on the clipboard trail, the "visited" flag propagates to the matching trail in the latest list asynchronously (gated by `CONST.detectPendingChangesInClipboard`, set to 300ms in the test). The opacity assertion (`≈ 0.6`) runs before that propagation completes, so the latest trail is still at full opacity (`1`). Expected to confirm by tracing the visited-state propagation timing vs. the assertion point.

**Steps**

*Phase 1 — Reproduce & instrument (parallelizable)*
1. Establish a local repro loop: run the karma suite repeatedly (e.g. `grunt karma:static` in a loop, or temporarily set `singleRun` and re-run N times) to get an empirical flake rate per test before any change.
2. Add temporary tracing (behind a debug flag, removed before final):
   - In [base-widget.js](public/src/js/widgets/base-widget.js#L9) / `getAutocomplete`, log which widget emits `widget:load` and the DOM state of `<autocomplete>` at that moment (confirms the race for failure 1).
   - In [base-action.js](public/test/utils/actions/base-action.js#L13), on timeout log whether the mockjax interceptor was ever hit and the last request seen (distinguishes "request never sent" vs "response callback never fired" for Sig B).
   - In [wait.js](public/test/utils/wait.js#L26) `condition`, on timeout log elapsed time and final predicate inputs (pinpoints which `check` stalls for Sig A). *(Already added — trace present in logs.)*
   - In [collections.spec.js](public/test/spec/collections.spec.js#L324) `copy to clipboard`, log the latest list trail count/contents at copy time and the clipboard state at test start (confirms Sig D race vs. state leak).
   - In [visited.articles.spec.js](public/test/spec/visited.articles.spec.js#L31), log the visited-state propagation timing relative to the opacity assertion (confirms Sig E timing).

*Phase 2 — Diagnose* (*depends on Phase 1*)
3. Using the traces from Phase 1, confirm or refute each of H1–H5 above, and record the precise failing synchronization point for each. (H1 confirmed; H2 disproven and replaced by H2' — awaiting image-load traces to confirm.)

*Phase 3 — Targeted fixes* (*depends on Phase 2*)
4. **Sig C (searches articles)**: Make `getAutocomplete` wait for the specific autocomplete widget rather than the first `widget:load`. Either filter the `widget:load` event by the emitting widget/element, or poll with `wait.condition` until `querySelector('autocomplete').firstChild` exists and has a ko context before reading `$component`.
5. **Sig B (both breaking-news tests)**: Replace/raise the fixed 2000ms in [base-action.js](public/test/utils/actions/base-action.js#L13) with a deterministic wait and ensure the paste→edit chain actually triggers the request before `execute()` awaits it; verify the `onAfterComplete` 20ms inner `setTimeout` isn't being cleared by teardown. This single fix should resolve both `Alerts the user before launch` and `Allows the same alert to be sent multiple times`.
6. **Sig A (create fronts and collections)**: Make the `changeImageUrl` step's image-validate path complete deterministically **without reducing coverage** (do **not** mock the image away). Pending H2' confirmation from the image-load traces, candidate fixes: ensure the real fixture image is preloaded/warm before the step, fix the flaky `onload`/`onerror` handling so the success signal reliably arrives, and/or make `wait.condition` and `jasmine.DEFAULT_TIMEOUT_INTERVAL` non-colliding. Do **not** simply raise `wait.condition` to 10000ms — that collides with Jasmine's 10s budget and leaks a polling interval (see H2' side effect).
7. **Sig D (copy to clipboard)**: Ensure the latest list is fully populated before resolving `trail(5)` (await a deterministic "latest loaded" signal), and assert/clear clipboard isolation in `beforeEach`/`afterEach` so state can't leak between specs.
8. **Sig E (visited articles)**: Await the visited-state propagation deterministically (a real signal that the latest list trail has been re-flagged) rather than reading opacity immediately after `openLink()`; if `CONST.detectPendingChangesInClipboard` gates it, wait on that completion explicitly before asserting.

*Phase 4 — Verify & clean up* (*depends on Phase 3*)
9. Remove temporary tracing (or gate it behind a permanent debug switch if broadly useful).
10. Re-run the repro loop to confirm the flake rate drops to zero for each test.

- **Sig A — FIXED (pending loop confirmation).** Root cause is an order-dependent global-singleton leak, **not** a timing budget (H2 disproven) **nor** a slow/failed image load (H2' disproven — `fetchImage onload after 0 ms`). `grid()` ([utils/grid.js](public/src/js/utils/grid.js#L7)) memoizes `grid.gridInstance`, reading `defaults.baseUrls.apiBaseUrl` only on first construction; the `config.front.spec` model lacked `baseUrls`, so when `changeImageUrl` was first to call `grid()` it threw `reading 'apiBaseUrl'` → error branch → no `saveProps` → timeout. **Fix:** added `baseUrls: { apiBaseUrl, mediaBaseUrl }` to the test model's `defaults` so `grid()` constructs deterministically regardless of spec order — keeps the full real validate/grid/recordUsage path (no mocking, no coverage loss). Reverted two earlier wrong attempts (the `, 10000` budget bump and the `persistence.front.update` callFake wait). **Latent follow-up:** the `grid.gridInstance` singleton still leaks across specs (built once, cached forever) — worth resetting in teardown as shared-utility hardening.
- **Sig B — FIXED & CONFIRMED (batch-12: 0 Sig B failures).** Two distinct sub-modes, both addressed; the `isPasteActive` subscription fix eliminated Sig B entirely from batch-12 (all 4 logs were Sig E).
  - **Sub-mode (b) — resolution lost (fixed):** the 2000ms reject timer in [base-action.js](public/test/utils/actions/base-action.js) used to start at **construction** (`actions.edit(...)`) instead of when the request fires in `execute()`. Moved the timer into a `startTimeout()` called at the start of `execute()`, with `clearTimeout` on resolve and in `dispose()`. Closed sub-mode (b).
  - **Sub-mode (a) — paste no-op, TRUE root cause (CONFIRMED batch-11, fixed):** the `.pasteOver` separator is gated by `css: {'pasteOver': $root.isPasteActive && $root.isPasteActive()}` ([collection.html](public/src/js/widgets/collection.html#L166), [trail.html](public/src/js/widgets/trail.html#L48)). `$root` is the top `BaseModel`; `isPasteActive` was defined **only** by the `copyPasteArticles` extension. Per [main.scala.html](app/views/templates/main.scala.html#L6), the column templates (`<main>`) bind **before** the sibling `extensions` foreach, and the extension is a KO **component whose viewModel is async-loaded via jspm**. On the **first** instantiation in a karma run (cold jspm cache), the column separators bind while `$root.isPasteActive` is still `undefined`; the `&&` short-circuits, so KO reads no observable and **creates no subscription**. The component viewModel then loads and defines `isPasteActive`, but the already-bound separators are never notified → `.pasteOver` never applies → `pasteItem` never runs → no `/edits` POST → 2000ms timeout. On later tests the viewModel is jspm-cached and loads synchronously, so the observable exists at bind time and it works — exactly matching the observed "only the first breaking-news test after a fresh load fails" pattern. **Batch-11 proved it conclusively:** the earlier product fix (init from `copiedArticle.peek()`) made `$root.isPasteActive()` read `true` at paste time, yet `.pasteOver` *still* never rendered (`condition timeout, predicate false`) — definitive evidence that the css binding had **no live subscription** to the observable.
    - **Fix (final):** declare `this.isPasteActive = ko.observable(false)` on [BaseModel](public/src/js/models/base-model.js) in its constructor, **before** any template binds, so every separator's css binding subscribes to a real, stable observable. The `copyPasteArticles` extension now **updates** that observable (`baseModel.isPasteActive(!!copiedArticle.peek())` + the `change` listener) instead of **replacing** it (which would have orphaned existing subscriptions); it keeps a defensive `ko.isObservable` guard. No mocking, full coverage retained. Leak-safe: `copiedArticle.flush()` runs in the BaseModel constructor before columns bind.
    - **Test-side hardening (kept):** `pasteOver()` in [group.js](public/test/utils/regions/group.js) / [trail.js](public/test/utils/regions/trail.js) waits via `wait.condition(() => $('.pasteOver', dom).length > 0)` before clicking, with a `group.pasteOver ENTER` diagnostic (separator? `$root.isPasteActive()`? clipboard peek?) — this diagnostic is what isolated the missing-subscription cause.
- **Sig D — ROOT CAUSE = clipboard storage leak (CONFIRMED batch-13).** Batch-13 trace is decisive: `latest trail count: 5` (correct) and `trail(5) headline: 'Nothing happened for once'` (the copy itself works), but **`clipboard trail count at start: 6`** (expected 0). So the clipboard already holds 6 leftover items at test start; `clipboard().trail(1)` returns the **oldest leaked item** (`'Open Source'`) instead of the just-copied article → assertion mismatch. It's a cross-test **state leak**, not a render race (H4 refined). Suspect mechanism: the clipboard storage key is built from a module-level `classCount` that **increments on every construction and only decrements on dispose** ([clipboard.js](public/src/js/widgets/clipboard.js#L18)); if clipboard instances ever overlap across tests the keys diverge and stale localStorage entries accumulate, and the 300ms `saveInStorage` poller can also write back during the async dispose gap. Added a `clipboard.getItemsFromStorage` trace (storage key, `classCount`, stored count) to pin the exact key/leak path next batch.
- **Sig E — copyToClipboard fix WORKED; new sub-cause found (batch-13).** The earlier no-op was fixed: batch-13 shows `openLink - href targets found: 1` (the clipboard trail now renders and the href IS clicked, so `setVisitedToTrue` fires `set:article:to:visited`). **But** `trailInLatest opacity` is *still* `1` and the test runs fast — so the emit isn't marking the matching **latest** article visited. Hypothesis: the id emitted from the clipboard article (`this.id()` in [article.js](public/src/js/models/collections/article.js#L165)) doesn't equal the latest article's `icc`, so [setArticleToVisited](public/src/js/models/collections/latest-articles.js#L140)'s `_.find` never matches (or a latest re-render replaces the marked instance). Added a `setArticleToVisited` trace (emitted id, matched?, latest ids) to confirm id-mismatch vs re-render next batch. Note the sibling test `marks an article visited from visited articles list` (opens the *latest* trail directly) passes — consistent with a clipboard-vs-latest id mismatch.
